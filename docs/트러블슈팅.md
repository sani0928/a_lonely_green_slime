## 1. 발생했던 문제

### 1.1 증상

- **플레이 시간이 약 20분을 넘어가면** 게임이 심하게 버벅이거나 사실상 멈춘 것처럼 동작함.
- 목표 플레이 타임(30분)까지 플레이하는 것이 불가능한 수준의 프레임 드랍 발생.
- 일부 환경에서는 **7분대 이후부터 몬스터가 전혀 스폰되지 않고**, 필드에는 플레이어만 남는 현상도 보고됨.

### 1.2 환경·제약

- **엔진**: Phaser 3 (2D 게임)
- **장르**: 생존형 액션 (맵 위 적 다수 스폰, 플레이어 추적, 자동 공격·충돌 처리)
- **월드**: 3200×3200 고정 맵, 카메라가 플레이어 추적
- **적**: 스폰 간격 280~650ms, 시간/플레이어 성장에 따라 간격 단축·추가 스폰 수 증가

## 2. 원인 분석

### 2.1 20분대 렉의 원인

원인을 **코드·데이터 흐름** 기준으로 정리하면 아래와 같다.

| 구분 | 내용 |
|------|------|
| **① 동시 존재 적 수 상한 없음** | 스폰 로직에는 “한 번에 몇 마리 더 뽑을지”만 제한(EXTRA_ENEMIES_CAP 4~6)이 있고, **전체 필드에 존재하는 적의 총 개수에 대한 상한이 없었음**. 스폰 간격이 280~380ms까지 짧아지므로, 20분이 지나면 적이 **수백 마리** 단위로 누적됨. |
| **② 매 프레임 O(n²) 연산** | `moveEnemiesTowardsPlayer`에서 **적 한 마리마다 “전체 적”을 순회하며 separation(밀집도 회피)을 계산**하고 있었음. n=200이면 프레임당 약 4만 번, n=300이면 약 9만 번의 비교·연산이 발생. |
| **③ 부가 비용** | 셀(구형 탄환) 타겟팅 `getNearestEnemyInRange`가 **탄환 개수 × 적 수**만큼 매 프레임 호출되고, 체인 공격 시에도 적 전체를 순회. 메모리 누수보다 **객체 수 증가 + O(n²) 로직**이 주원인으로 판단됨. |

즉, “렉”의 직접 원인은 **n이 무한정 커질 수 있는 구조**와 **n에 대해 이차로 늘어나는 연산**이 결합된 것이었다.

### 2.2 스폰 정지 버그의 원인 (최적화 후 발견)

- 최적화로 **동시 존재 적 수 상한(예: 300)** 을 도입한 뒤, “필드에 몬스터가 없는데 7분쯤 지나면 스폰이 멈춘다”는 현상이 보고됨.
- 스폰 조건이 `countActive(true) >= MAX_ACTIVE_ENEMIES` 일 때 **무조건 return** 하는 구조였음.
- 플레이어가 한 방향으로 계속 이동하면, 적들은 플레이어를 따라가느라 **화면 뒤쪽(월드 반대편)** 에 몰려 있게 됨.
- 이때 **화면에는 안 보이지만**, 그룹에는 여전히 300마리가 “활성”으로 잡혀 있어서 **countActive(true)가 300을 유지** → 스폰이 영구히 막힘.
- 따라서 “몬스터가 안 나온다”는 현상은 **상한만 두고, “멀리 있는 적을 어떻게 할지”를 고려하지 않은 설계**에서 비롯된 부작용이었다.

## 3. 해결 과정

### 3.1 적 수 상한 도입 (필수)

- **목적**: n이 무한히 커지지 않도록 상한을 두어, 최악의 연산량을 제한.
- **구현**:
  - `constants.js`에 `MAX_ACTIVE_ENEMIES_BASE`, `MAX_ACTIVE_ENEMIES_LATE` 및 램프용 상수 추가.
  - `spawnEnemy` 진입 시 `getMaxActiveEnemies(scene)`으로 현재 허용 상한을 구한 뒤, `countActive(true) >= cap` 이면 즉시 return.
  - “추가 스폰” 루프 안에서도 매번 상한을 다시 확인하고, 초과 시 break 하여 한 틱에 상한을 넘지 않도록 함.
- **밸런스**: 초반은 300 수준으로 두고, 후반으로 갈수록 400~500 수준까지 완만히 올리는 **시간 기반 상한**을 적용해, 난이도 곡선과 성능을 함께 고려함.

### 3.2 Separation 연산을 공간 해시로 O(n²) → O(n×k) 축소 (핵심)

- **기존**: `moveEnemiesTowardsPlayer`에서 **적 1마리당 전체 적 리스트**를 도는 이중 루프 → O(n²).
- **개선**:
  - **그리드 기반 공간 해시**를 프레임당 **한 번만** 구성: `buildEnemyGrid(scene, cellSize)` → `Map<"cx,cy", Enemy[]>`.
  - 셀 크기 64px, separation 반경(52)의 약 2배 수준으로 두어, **자신이 속한 셀 + 인접 8셀** 안의 적과만 separation 계산.
  - 한 셀당 적 수를 k라 하면 비용은 O(n×k). k를 작게 유지해 n² 대비 비용을 크게 줄임.
- **구현 포인트**:
  - `getNeighborsForSeparation(enemy, grid, cellSize)`로 해당 적이 속한 셀 + 인접 8셀의 적만 반환.
  - `moveEnemiesTowardsPlayer`는 `buildEnemyGrid` 결과(또는 scene에 캐시된 그리드)를 받아, separation 시 그리드만 사용하고 `getChildren()` 중복 호출 제거.

### 3.3 셀 탄환 타겟팅 비용 축소 및 그리드 1회 생성

- **기존**: `getNearestEnemyInRange`가 **탄환마다** 전체 적을 순회 → O(탄환 수 × n).
- **개선**:
  - 그리드는 **GameScene.update에서 1회만** 생성.
  - `EnemySystem.buildEnemyGrid(this, 64)` → `scene._enemyGrid`에 저장(또는 반환값 유지).
  - `moveEnemiesTowardsPlayer(this, enemyBuilt)`에서 separation은 이 그리드 사용.
  - `CellSystem.updateCellProjectiles(this, dt)`에서 `getNearestEnemyInRange`가 `scene._enemyGrid`(또는 전달된 grid)를 사용해 **(x, y) 근처 셀만** 검사 → O(탄환×k) + 그리드 1회 O(n).

이렇게 해서 **그리드 생성은 프레임당 1회**, separation과 셀 타겟팅이 **동일한 자료구조**를 재사용하도록 정리했다.

### 3.4 정리

- **상한 도입**으로 n의 상한을 고정하고,
- **공간 해시**로 separation·타겟팅 비용을 O(n²), O(탄환×n)에서 O(n×k), O(탄환×k) 수준으로 낮추고,
- **그리드 1회 생성**으로 중복 연산과 일관성을 맞추는 흐름으로 해결했다.

## 4. 추가로 발견·해결한 이슈

### 4.1 몬스터 스폰 정지 (화면 밖 300마리 쌓임)

- **원인**: 상한 300만 있어서, 화면 밖에 300마리가 쌓이면 `countActive(true) >= 300` 이 되어 스폰이 영구히 막힘.
- **해결**:
  - **원거리 적 컬링** 도입: 플레이어와의 거리가 `ENEMY_CULL_DISTANCE`(1400px)를 넘는 적은 매 프레임 제거(`cullDistantEnemies`).
  - GameScene.update에서 난이도·이동 처리 직후, 그리드 구성 직전에 `EnemySystem.cullDistantEnemies(this)` 호출.
- **효과**: “화면 밖에만 300마리”가 쌓여 스폰이 막히는 상황이 사라지고, 플레이어 주변으로만 적이 유지되어 스폰이 정상 동작함.

### 4.2 후반부 난이도 곡선 (상한 고정으로 인한 지루함)

- **문제**: 상한을 300으로 고정하니 중후반에 “적 수”가 일정해져 도전감이 떨어짐.
- **해결**:
  - **시간에 따른 적 상한 램프**: `getMaxActiveEnemies(scene)`에서 `elapsedTime`에 따라 BASE(300) → LATE(400~500)로 완만히 상승.
  - 램프 시작/종료 시점(`ENEMY_CAP_RAMP_START_SEC`, `ENEMY_CAP_RAMP_END_SEC`)으로 6분~24분 구간에서 선형 보간.
  - **후반 스폰 간격 추가 단축**: 10분 이후 `MIN_SPAWN_DELAY_MS_LATE`(250ms) 적용으로, 상한이 올라갈 때 채워지는 속도를 높여 압박감 유지.

## 5. 결과 및 배운 점

### 5.1 정량적 결과 (요약)

| 항목 | 최적화 전 | 최적화 후 |
|------|-----------|-----------|
| 동시 적 수 | 무제한(수백 마리 누적) | 상한 300 → 후반 400~500 (시간 램프) |
| Separation 비용 | O(n²) 매 프레임 | O(n×k), k는 셀당 적 수 |
| 셀 타겟팅 | O(탄환 수 × n) | O(탄환 수 × k) + 그리드 1회 O(n) |
| 20분대 플레이 | 심한 프레임 드랍·정지 | 목표: 30fps 근처 유지 |
| 스폰 정지 버그 | (상한 도입 후 발생) | 원거리 컬링으로 해소 |

### 5.2 설계·트러블슈팅에서 얻은 점

- **상한 없는 리스트 + 매 프레임 O(n²)** 조합이 장시간 플레이에서 치명적일 수 있음을 확인했고, **상한 도입**과 **연산 차수 축소**를 동시에 적용하는 것이 중요했다.
- **“보이지 않는 객체”가 게임 로직(스폰 조건)에 영향을 주는 경우**를 간과했고, 상한만 두면 “화면 밖에 쌓인 300마리”로 인해 스폰이 막힐 수 있음을 배웠다. **컬링(거리 기반 제거)** 으로 “실제로 의미 있는 적”만 남기고, 상한은 “의미 있는 적”에만 적용되도록 정리했다.
- **공간 해시(그리드)** 를 한 번 만들고 separation·타겟팅이 같이 쓰도록 하여, **구조 재사용**과 **연산 횟수 감소**를 동시에 달성했다.
- 난이도는 “상한 고정”만으로는 부족하고, **시간에 따른 상한 램프**와 **후반 스폰 간격 조정**을 함께 두어 “중후반이 지루해지는” 문제를 완화했다.

### 5.3 기술 스택·역할 정리 (포트폴리오용)

- **Phaser 3**: 2D 게임 루프, 물리 그룹, 스폰/업데이트 로직.
- **성능**: 연산 복잡도 분석(O(n²) → O(n×k)), 공간 분할(그리드 기반 공간 해시), 상한·컬링으로 객체 수 제어.
- **게임 밸런스**: 적 수 상한, 시간 기반 램프, 스폰 간격·컬링 거리 튜닝.

이 문서는 위 트러블슈팅을 **문제 → 원인 → 해결 과정 → 추가 이슈 → 결과·교훈** 순으로 정리한 포트폴리오용 기록이다.


요약

n: 동시에 존재하는 몬스터 수.
separation: 몬스터가 서로 너무 겹치지 않도록 밀집도를 피하는 연산.
이전 문제: n에 상한이 없어 무한히 커질 수 있고, separation이 한 몬스터당 전체 몬스터(n마리) 와 비교 → 프레임당 O(n²) 로 렉 발생.
해결:
1) n 상한으로 n이 무한히 커지지 않게 함.
2) 공간 해시(그리드) 로, 한 몬스터당 자신이 속한 셀 + 인접 8셀(총 9셀) 안에 있는 몬스터(k마리)와만 separation 계산 → 프레임당 O(n×k) 로 비용 감소.
여기서 k는 “그 9개 셀 안에 있는 몬스터 수”라서 몬스터마다 다르고, 보통 n보다 훨씬 작기 때문에 n×k가 n²보다 크게 줄어듬

셀 타겟팅 = “각 셀(구형) 탄환이 누구를 타겟으로 할지 정하기 위해 가장 가까운 적을 찾는 연산”.
이전: 탄환마다 전체 적 n을 순회 → O(탄환 수 × n).
이후: separation용으로 만든 그리드를 재사용해서, (cellX, cellY) 반경 안의 몇 개 셀 안의 적 k마리만 검사 → O(탄환 수 × k) 로 비용 감소.
